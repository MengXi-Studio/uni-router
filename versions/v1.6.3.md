# @meng-xi/uni-router v1.6.3 更新：修复 H5 端 interceptUniApi 导致 TabBar 无法点击

> v1.6.3 修复了启用 `interceptUniApi` 后 H5 端 TabBar 菜单无法点击的问题，同时恢复了 `switchTab` 的路由守卫能力

## 前言

`interceptUniApi` 是 uni-router 的重要特性，启用后会拦截 `uni.navigateTo` / `uni.redirectTo` / `uni.switchTab` / `uni.reLaunch` / `uni.navigateBack`
五个原生导航 API，将外部直接调用重定向到路由器实例，确保路由守卫始终生效。

然而在 H5 平台下，启用 `interceptUniApi: true` 会导致 TabBar 上的菜单无法点击，而小程序端表现正常。将 `interceptUniApi` 设为 `false` 后，TabBar 即可正常点击。

v1.6.3 修复了这一问题，并恢复了 `switchTab` 走守卫链的能力，使得在守卫中拦截 TabBar 切换（如未登录时跳转个人中心）成为可能。

---

## 一、问题分析

### 现象

- H5 端：`interceptUniApi: true` 时，点击 TabBar 菜单无任何反应
- 小程序端：`interceptUniApi: true` 时，TabBar 正常工作
- `interceptUniApi: false` 时，两端均正常

### 根因

拦截器 `invoke` 钩子中的执行顺序存在 bug：

```typescript
// 修复前：interceptor/index.ts
invoke(args: Record<string, any>) {
    if (activeManager?.isRouterCall()) {
        return args
    }
    // ❌ 先清空 URL
    if ('url' in args) args.url = ''
    // ❌ 再处理导航 → parseUniUrl('') 返回空路径，router.push 永远不会执行
    return handleInterceptedNavigation(api, args)
}
```

`args.url = ''` 在 `handleInterceptedNavigation()` **之前**执行，导致 `parseUniUrl('')` 返回空路径 `{ path: '', query: {} }`，`if (path)` 判断为 `false`，`router.push` 永远不会被调用——switchTab 导航被完全吞掉。

### 为什么小程序端正常

这是理解此 bug 的关键：

- **小程序端**：TabBar 是原生组件，点击 Tab 时由原生层直接处理页面切换，**不经过 `uni.switchTab`**，拦截器根本不会被触发
- **H5 端**：TabBar 是 Vue 组件，点击时调用 `uni.switchTab`，拦截器触发后 URL 被提前清空，导航被吞掉

所以小程序端的"正常"并非拦截器工作正确，而是拦截器根本没被触发。

---

## 二、修复方案

调换执行顺序：**先解析 URL 并触发路由器导航，再清空 URL 作为双重保险**：

```typescript
// 修复后：interceptor/index.ts
invoke(args: Record<string, any>) {
    if (activeManager?.isRouterCall()) {
        return args
    }
    // ✅ 先处理拦截逻辑（在 URL 被清空前解析路径并触发路由器导航）
    const result = handleInterceptedNavigation(api, args)
    // ✅ 再清空 URL 防止低版本基础库忽略返回值
    if ('url' in args) args.url = ''
    return result
}
```

同时恢复 `switchTab` 在 `handleInterceptedNavigation` 中的处理：

```typescript
case 'switchTab': {
    const { path } = parseUniUrl(args.url || '')
    if (path) {
        router.push(path)
    }
    break
}
```

所有五个导航 API（`navigateTo` / `redirectTo` / `switchTab` / `reLaunch` / `navigateBack`）统一使用同一个拦截器注册逻辑，不再对 `switchTab` 做特殊处理。

---

## 三、switchTab 守卫能力恢复

修复前，由于 `switchTab` 的 URL 被提前清空，`router.push` 从未被调用，守卫链根本不会执行。修复后，`switchTab` 正确走 `router.push` → 守卫链 → `uni.switchTab` 的完整流程。

这意味着你可以在守卫中拦截 TabBar 切换，例如：

```typescript
// 典型场景：首页不需要登录，但个人中心需要
router.beforeEach((to, from, next) => {
	if (to.meta.requireAuth && !isLoggedIn()) {
		// ✅ switchTab 到个人中心时，守卫可以正确拦截并重定向到登录页
		next({ name: 'login', query: { redirect: to.fullPath } })
	} else {
		next()
	}
})
```
