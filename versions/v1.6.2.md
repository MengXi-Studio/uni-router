# @meng-xi/uni-router v1.6.2 更新：isReady() 执行时机修正

> v1.6.2 修复了 `isReady()` 在 `createRouter()` 构造时即触发的问题，将其推迟到 `app.use(router)` 安装阶段，确保回调中可安全使用所有已安装插件

## 前言

v1.5.0 引入了 `readyTimeout` 超时保护机制，防止 `isReady()` Promise 永久挂起。但在实际使用中发现，`isReady()` 的就绪触发时机存在一个微妙但重要的问题：

**`markReady()` 在 `setCurrentRoute()` 首次调用时触发**，而 `setCurrentRoute()` 在 `createRouter()` 构造函数中即被调用（通过 `initCurrentRoute` 初始化起始路由位置）。这意味着 `isReady()` 的回调会在 `app.use(router)`
之前就执行，此时其他插件（如 Pinia）尚未安装，回调中无法安全使用这些插件。

v1.6.2 将 `markReady()` 从 `setCurrentRoute()` 中移出，改为在 `install()` 方法中显式调用，确保 `isReady()` 回调在所有插件安装完成后执行。

---

## 一、问题分析

### 原始行为

```typescript
// 优化前：state/index.ts
function setCurrentRoute(route: RouteLocation): void {
	// ...设置路由对象...

	if (!ready) {
		ready = true
		if (readyTimer) {
			clearTimeout(readyTimer)
			readyTimer = null
		}
		for (const resolve of readyResolvers) {
			resolve()
		}
		readyResolvers.length = 0
		readyRejecters.length = 0
	}
	// ...
}
```

`setCurrentRoute()` 在 `createRouter()` 构造函数中通过 `initCurrentRoute()` 被首次调用，此时路由器刚刚创建，`app.use(router)` 尚未执行。

### 问题场景

```typescript
// main.ts
const router = createRouter({ routes }) // ← 此时 markReady 已触发

// 以下代码在 createRouter 期间就已执行
router.isReady().then(() => {
	// ❌ 此时 Pinia 尚未安装，useStore() 会失败
	const store = useUserStore()
})

const pinia = createPinia()
const app = createSSRApp(App)
app.use(pinia) // ← Pinia 在此安装
app.use(router) // ← 路由器在此安装
```

在上述场景中，`isReady()` 的 Promise 在 `createRouter()` 时就已经 resolve，回调中的 `useUserStore()` 会因为 Pinia 尚未安装而失败。

---

## 二、修复方案

将 `markReady()` 从 `setCurrentRoute()` 中提取为独立方法，在 `install()` 中显式调用：

```typescript
// 修复后：state/index.ts — markReady 提取为独立方法
function markReady(): void {
	if (ready) return
	ready = true
	if (readyTimer) {
		clearTimeout(readyTimer)
		readyTimer = null
	}
	for (const resolve of readyResolvers) {
		resolve()
	}
	readyResolvers.length = 0
	readyRejecters.length = 0
}

function setCurrentRoute(route: RouteLocation): void {
	// ...设置路由对象...
	// 不再在此处调用 markReady
}
```

```typescript
// 修复后：router/index.ts — install() 中显式调用 markReady
install(app: App) {
    // ...注册 $router、$route、provide 等...

    // 在 install 时标记路由器就绪，确保 isReady() 回调在所有插件安装完成后执行
    this.routeState.markReady()
}
```

### 修复后的行为

```typescript
// main.ts
const router = createRouter({ routes }) // ← markReady 不再触发

router.isReady().then(() => {
	// ✅ 此时 Pinia 已安装，useStore() 可正常使用
	const store = useUserStore()
})

const pinia = createPinia()
const app = createSSRApp(App)
app.use(pinia)
app.use(router) // ← markReady 在此触发，所有插件已就绪
```

## 三、readyTimeout 超时保护

当配置了 `readyTimeout > 0` 时，如果 `install()` 在超时时间内未被调用（如路由器未通过 `app.use()` 安装），`isReady()` 将 reject 并抛出超时错误：

```typescript
const router = createRouter({
	routes,
	readyTimeout: 5000 // 5秒超时
})

// 如果 5 秒内未调用 app.use(router)，isReady() 将 reject
router.isReady().catch(err => {
	console.error('路由器初始化超时:', err)
})
```

## 升级指南

v1.6.2 是 Bug 修复，**完全向后兼容**，无需修改任何现有代码即可升级。

### 行为变化

| 场景                     | v1.6.1 及之前            | v1.6.2                                                |
| ------------------------ | ------------------------ | ----------------------------------------------------- |
| `isReady()` 触发时机     | `createRouter()` 构造时  | `app.use(router)` 安装时                              |
| 未调用 `app.use(router)` | `isReady()` 立即 resolve | `isReady()` 挂起（配置 `readyTimeout` 后超时 reject） |
| 回调中使用其他插件       | ❌ 可能失败              | ✅ 安全可用                                           |

如果你之前在 `isReady()` 回调中依赖了早期触发行为，升级后回调执行时机将推迟到 `app.use(router)` 之后。对于绝大多数场景，这正是期望的行为。
